# V3.46 — Compliance: + Submit Certificate flow + retire `/apps/compliance/`

## Goal

Fold the lab-app student-submission flow at `/apps/compliance/` into
[rm/pages/compliance.html](../../rm/pages/compliance.html). Same data
shape (`complianceSubmissions` collection, same fields) so the existing
admin-side Student Training tab keeps working unchanged. Once the modal
ships, delete the lab app — students click the same `+ Submit
Certificate` button on the RM compliance page; admins keep verifying
from the Student Training tab as before.

This is the same audit-gap pattern V3.44 surfaced for purchases /
procurement — the RM admin pages were missing student-submission flows.
Here the gap is small (one modal, one collection, no new page), so V3.46
both adds the flow and deletes the lab app in a single release rather
than splitting across two phases like V3.44 / V3.45 did.

## What changes for the user

- **Lab member (non-admin):** opens [rm/pages/compliance.html](../../rm/pages/compliance.html),
  hits the new `+ Submit Certificate` button (visible on the Student
  Training tab), uploads cert PDF/image, fills type + title + date +
  optional expiry, submits. Their own submissions show in the Training
  tab list (filtered server-side by `where('submittedBy','==',uid)`).
  The `+ Add Protocol` button is hidden on the IRB/IACUC tabs — those
  collections are admin-write per `firestore.rules`.
- **Admin / PI:** same page, but `+ Add Protocol` (IRB/IACUC tabs) and
  `+ Submit Certificate` (Training tab) both work. Training tab uses
  `firebridge.getAll('complianceSubmissions', ...)` (full collection
  read; admin rules permit). Verify button still appears on rows with
  `status: 'submitted'`.

The page-header button text and behavior are now contextual:

| Active tab | Admin sees | Non-admin sees |
|---|---|---|
| IRB / IACUC | `+ Add Protocol` (opens existing protocol form) | hidden (rule = admin-only write) |
| Student Training | `+ Submit Certificate` (opens new modal) | `+ Submit Certificate` (opens new modal) |

## Implementation

### `rm/js/compliance.js`

Three additions on top of the existing protocol-tracking renderer:

1. **`_isAdmin()` helper + `_updateAddButton()` toggler.** Called from
   `loadAndRender()` on every tab switch. Updates the `#add-item`
   button's text + visibility based on `(activeTab, role)`.
2. **`renderTrainingTab()` per-user fork.** Admin path keeps the
   existing `firebridge.getAll('complianceSubmissions', ...)`. Non-admin
   path does
   `firebridge.db().collection('complianceSubmissions').where('submittedBy','==',uid).orderBy('createdAt','desc').get()`
   — required because `firestore.rules` line 478 restricts read to
   `(own || isAdmin)` and a bare getAll would fail per-doc rule checks
   on any doc the user doesn't own. Empty state is friendlier for
   non-admins ("…click + Submit Certificate above to upload your first
   one"). The Verify button is gated on `_isAdmin()` so non-admins
   don't see a button that would be permission-denied at write time.
3. **`openSubmitCertModal()`.** Builds a modal directly (not via
   [forms.js](../../rm/js/forms.js) because that doesn't support file
   inputs). Validates type + title + completion date. On submit:
   uploads the file to `compliance/{uid}/{Date.now()-rand}/<safeName>`
   (Storage rule at [storage.rules:78](../../storage.rules#L78) — auth
   + 10 MB + PDF or image), then writes to `complianceSubmissions` with
   `status: 'submitted'`, `storagePath`, `certificateUrl`, and the
   form fields. After save, switches `activeTab = 'training'` and
   re-renders so the user immediately sees their new row.

### `rm/pages/compliance.html`

Adds `firebase-storage-compat` to the SDK loads (the page never used
Storage before V3.46). Comment in place to flag why.

### `lab-apps.js`

`compliance` entry removed from `LAB_APPS` registry. Registry was
already orphaned in the public nav since V3.40; consistency matters
because Phase C will delete the file outright.

### `apps/compliance/`

Deleted. Same shape as V3.42 (console) and V3.43 (inventory) — the
3-file directory (app.js + index.html + styles.css) goes via
`git rm -rf`. The two paragraphs of pre-existing local cache-buster
modifications (`?v=2` on the index.html) discard with the directory.

### `firestore.rules` / `storage.rules`

**Both unchanged.** The existing `complianceSubmissions` rule at line
478 already permits the access pattern V3.46 needs (auth-create,
read-own-or-admin, admin-update). The Storage rule for
`compliance/{userId}/**` at line 78 already permits owner-or-auth
upload of PDF/image up to 10 MB. No deploy needed.
