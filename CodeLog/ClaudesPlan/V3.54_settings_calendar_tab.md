# V3.54 — Settings: Calendar Sync tab

## Goal

Close the V3.48 deferred work: lift the lab-app `/apps/settings/`
Calendar Integration UI into [rm/pages/settings.html](../../rm/pages/settings.html)
as a new **Calendar Sync** tab. With the V3.51 CalendarService now
loaded in RM, users can manage Google / Outlook / Apple calendar
connections from the Settings page instead of having to fall back
to the (now-deleted) standalone lab app.

## Tab placement

`Calendar Sync` slots into the personal-config cluster at the front
of the tab bar, alongside Profile and Notifications:

```
[Profile] [Notifications] [Calendar Sync] [All Connections (n)] [Email (n)] [Calendars (n)] ...
```

**Key collision dodge**: the existing connection-registry filter at
key `'calendar'` (third-party connection type filter, label
"Calendars") is unrelated to the new Calendar Sync UI. Renaming
either would break the connection-type filter logic
(`connections.filter(c => c.type === activeTab)` keys on the literal
`'calendar'` string in stored docs). The new tab uses key
`'cal-sync'` to keep both clear.

## What landed

### `rm/js/settings.js`

- New `cal-sync` entry in `TABS` array, positioned between
  `notifications` and `all`.
- `_isConnectionTab()` updated to also exclude `cal-sync` (so the
  `+ Add Connection` page-header button stays hidden on this tab).
- New `_isConnectionTabKey(k)` helper for the tab-bar label-counting
  logic — keeps the `if (k !== 'profile' && k !== 'notifications')`
  check from creeping further out of date as more personal-config
  tabs land.
- New dispatcher branch:
  `if (activeTab === 'cal-sync') return renderCalendarSync(content)`.
- New `renderCalendarSync(content)` function (~150 LOC). Three
  provider cards (Google / Outlook / Apple), sync settings (refresh
  interval + Sync to Huddle toggle), Save button + status pill.
  Adapted from the lab-app's `renderCalendarSection()` with the
  V3.41/V3.47/V3.49/V3.50/V3.52 RM-idiom pattern: `.cal-card` /
  `.btn` / `.btn-primary` / `.proc-btn-danger` / `.settings-toggle-row`
  / `.settings-status` (no lab-app `.settings-save-btn`).
- New `wireCalendarSync(cal)` function (~120 LOC). Adapted from the
  lab-app `wireCalendar()`. Same handlers:
  - Google: `cal.connectGoogle(clientId)` + `cal.disconnectGoogle()`.
  - Outlook ICS file/URL: `cal.importICSFile(file, 'outlook_ics')`,
    `cal.fetchICSFromUrl(url, 'outlook_ics')`, `cal.clearProvider`.
  - Outlook OAuth (advanced): `cal.connectOutlook(clientId)` +
    `cal.disconnectOutlook()`.
  - Apple ICS file/URL: same `importICSFile` / `fetchICSFromUrl` /
    `clearProvider` pattern with `'ics'` provider key.
  - Save sync settings: `cal.saveConfig({ autoRefreshMinutes,
    huddle: { autoBlock } })` + `cal.startAutoRefresh(refreshMinutes)`.
- Provider icons (`_googleIcon` / `_outlookIcon` / `_appleIcon`)
  inlined as SVG strings — same as the lab-app helpers.

### `rm/css/settings.css`

New `.cal-*` classes for the provider-card layout: `.cal-providers`
(grid auto-fit min 280px), `.cal-card`, `.cal-header`, `.cal-icon`,
`.cal-badge` + `.cal-badge--on`, `.cal-body`, `.cal-row`,
`.cal-actions`, `.cal-hint`, `.cal-details`, `.cal-select`. Plus a
local copy of `.proc-btn-danger` (lifted from
[rm/css/procurement.css](../../rm/css/procurement.css)) so the
"Disconnect" / "Clear" buttons get the red-tint danger state on the
settings page without depending on procurement.css being loaded.

### `rm/pages/settings.html`

Added `<script src="/rm/js/calendar-service.js">` between
`forms.js` and `settings.js` so `McgheeLab.CalendarService` is
defined by the time `settings.js` runs. The GIS client (already
loaded for the existing connection-registry OAuth handling) covers
`cal.connectGoogle()`'s GIS token-client requirement.

### `sw.js`

`CACHE_VERSION` bumped 28 → 29.

## Verification

Local dev:

```bash
cd /Users/alexmcghee/Documents/GitHub/McGheeLabWebsite
RM_WRITE_ALLOWLIST=mcgheealex@gmail.com python3 rm/server.py
# open http://localhost:8000/rm/pages/settings.html
```

Smoke flow:

1. Sign in as lab member → page lands on Profile tab. Click
   `Calendar Sync` → three provider cards + sync settings render.
2. Google card: paste an OAuth Client ID → click `Connect Google
   Calendar`. GIS token-client fires; on success the badge flips
   to "Connected" and the Connect button is replaced by Disconnect.
3. Apple card: paste a published iCloud calendar URL → click
   `Fetch`. ICS gets downloaded via the calendarProxy Cloud Function
   fallback; status pill shows "ICS fetched — N events".
4. Outlook card: same pattern. Use the .ics file upload OR paste
   the public ICS URL OR open Advanced and connect via OAuth.
5. Set auto-refresh to "Every 30 minutes", toggle "Sync to Huddle"
   on, click `Save Calendar Settings`. Status pill confirms.
   `cal.startAutoRefresh(30)` schedules background re-fetch every
   30 min while the page stays open.
6. Reload page → connection state survives (config persisted to
   `userSettings/{uid}.calendar`; tokens are in `sessionStorage`
   so re-OAuth needed only on a fresh browser session).
7. Switch to `All Connections` tab → `+ Add Connection` button
   appears in the page header (only on connection tabs);
   connection-registry table renders unchanged.
8. DevTools → Application → Cache Storage shows
   `mcgheelab-shell-v29` only.
