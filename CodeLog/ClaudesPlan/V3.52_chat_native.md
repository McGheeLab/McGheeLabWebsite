# V3.52 — Lab Chat: native RM port

## Goal

Replace the V3.40 iframe wrapper at
[rm/pages/app-chat.html](../../rm/pages/app-chat.html) with a native
renderer at [rm/pages/chat.html](../../rm/pages/chat.html). This is
the largest single port in the migration — 3,630 LOC + 1,852 LOC of
CSS — but it follows the same lift-and-patch playbook used for V3.41
meetings, V3.49 activity-tracker, and V3.50 huddle.

Chat does **not** require any V3.51 OAuth deferral. Drive uploads
flow through a Google Apps Script web app (URL stored in
`chatConfig.gdriveScriptUrl`) — the lab app explicitly notes "no
per-user token needed" at line 114 of `app.js`. So the file-attach
modal continues to work end-to-end without any bridge to the
deferred CalendarService / ScheduleService refactor.

## What ports

The full chat surface, unchanged:

- **Channel feed** — main message list with reactions (👍 / ❤️ / etc.),
  threads, @mentions, edit / delete, reply chain.
- **DM feed** — paired-uid channels in `chatChannels` with the
  `dmChannelIds` index on `chatUserMeta/{uid}`.
- **Thread panel** — open per-message thread on the right rail
  (desktop) or as a stack on top (mobile).
- **Channel directory + admin-defined categories** — read from
  `chatConfig.categories`, sorted; admin-only edit lives behind the
  chat-admin gate.
- **Sidebar with draggable custom groups** — user reorders channels
  into per-user buckets, persisted to `chatUserMeta/{uid}.sidebar`.
- **Browser push notifications** — requests permission on first @mention
  / DM, fires through the existing `users/{uid}/pushTokens` infra.
- **Files view** — groups channel attachments by type, lazy-loaded.
- **Mobile shell** — conversation-list-first state machine (`_mobileView`)
  with stats bar, search, files, hamburger settings panel.
- **Drive file attach** — uploads via `chatConfig.gdriveScriptUrl`
  Apps Script; legacy `driveUrl` field on old messages still
  resolves alongside the modern `url` (Firebase Storage) field.

## Patches applied to the lifted file

1. **`window.McgheeLab = window.McgheeLab || {}`** at top — same
   pattern as V3.49 / V3.50. Optional-chaining lookups for
   `MobileShell?.saveTabScroll` etc. need the namespace to exist.
2. **Container id `#app` → `#chat-root`** — avoids collision with
   other RM pages.
3. **`db()` prefers `firebridge.db()`** when available, falls back
   to `firebase.firestore()`. `TS()` (FieldValue.serverTimestamp)
   stays as-is.
4. **Bootstrap rewrite (lines 137–208 of the lab app)** — replaces
   the dual AppBridge / direct-Firebase / setInterval-poll path
   with the RM contract: `firebridge.gateSignedIn()` →
   `await firebridge.whenAuthResolved()` → `_user/_profile` from
   firebridge → `canAccessChat(_profile)` chat-specific guest gate
   → `loadConfig() / loadUsers() / loadUserMeta() / render()` →
   subscribes (channels, read-state, users, config) → `initChatTabSwipe()`.
   `firebridge.onAuth` handles role flips so newly-promoted chat
   admins see admin affordances without a manual reload.
5. **`notifyResize()` AppBridge guard** — patched so the
   `McgheeLab.AppBridge.isEmbedded()` call no-ops cleanly when
   AppBridge is undefined (RM is the parent window — no postMessage
   to fire).

The remaining ~3,500 LOC of message rendering, reaction UI, thread
fan-out, sidebar drag-drop, files view, mobile state machine, push
permission flow, and Drive upload integration transfers verbatim.

## Key decisions

- **No OAuth deferral.** Chat's Drive uploads use an Apps Script
  web app (`chatConfig.gdriveScriptUrl`), not per-user Google OAuth.
  The lab app explicitly calls this out at line 114 ("no per-user
  token needed"). So unlike scheduler / huddle / activity-tracker
  / settings, chat ports cleanly without waiting for V3.51.
- **Verbatim lift, not rewrite.** 3,500 LOC of view code is correct
  and battle-tested. Same call as every prior large port.
- **Chat-specific guest gate preserved.** `canAccessChat(_profile)`
  at the top of the boot fn lets admin enable chat for a guest user
  without flipping their role in `users/{uid}` — useful for
  external collaborators who join a single channel but aren't lab
  members.
- **No LIVE_SYNC.attach.** The renderer uses Firestore `onSnapshot`
  directly via `subscribeChannels` / `subscribeReadState` /
  `subscribeUsers` / `subscribeConfig`. Layering LIVE_SYNC over
  that would duplicate work; the existing approach is correct.
- **Mobile state machine retained.** The chat-specific
  conversation-list-first mobile UI (`_mobileView` state machine)
  is one of the most polished UX flows in the lab apps and ports
  unchanged.

## Verification

Local dev:

```bash
cd /Users/alexmcghee/Documents/GitHub/McGheeLabWebsite
RM_WRITE_ALLOWLIST=mcgheealex@gmail.com python3 rm/server.py
# open http://localhost:8000/rm/pages/chat.html
```

Smoke flow:

1. Sign in as guest → pending-access overlay (firebridge gate).
   Sign in as guest with chat enabled → chat loads.
2. Sign in as lab member → page renders with sidebar + welcome
   message in #general (or last-active channel).
3. Type a message + Enter → message appears live (own write echoes
   back via the `subscribeChannels` snapshot).
4. Open in second tab → mutate in tab 1 → tab 2 sees the new
   message within ~1s (snapshot-driven, no LIVE_SYNC needed).
5. Click a message → reaction picker → 👍 → reaction count updates.
6. Open a thread → reply → thread panel updates.
7. Send a DM → new channel under DMs section.
8. @mention a teammate → browser push permission prompt fires
   (first time) → notification arrives via existing `users/{uid}/pushTokens`
   path.
9. Click file-attach → upload to Drive (Apps Script) → URL
   embedded in the message; opens cleanly in a new tab.
10. Mobile (<700px viewport) → conversation-list-first layout.
    Tap a conversation → swap to conversation view. Hamburger →
    settings drawer.
11. `http://localhost:8000/rm/pages/app-chat.html` → meta-refreshes
    to `/rm/pages/chat.html`.
12. `http://localhost:8000/apps/chat/` → standalone fallback still
    loads.
13. DevTools → Application → Cache Storage → only `mcgheelab-*-v26`
    caches.
