# V3.6 — Drag-and-Drop Course Builder

## Goal

Replace the current section-based class builder (fixed sidebar + content area) with a drag-and-drop course builder page. Admin adds sections and widgets by dragging them from palettes into a page layout. Autosave every 30 seconds persists changes to Firestore without manual save buttons.

## Current State

- `class-scheduler.js` has a `SECTION_REG` with 9 section keys mapped to 3 component types (text, files, speakers)
- Admin uses Settings → checkbox list to toggle sections on/off, then clicks "Save Sections"
- Each section renders in a single `#class-section-content` area, swapped by sidebar nav clicks
- No autosave — every section has its own "Save" button

## New Workflow

1. **Admin Panel → Add Course Listing** — already auto-creates both `classes` doc + `schedules` doc (done in V3.5)
2. **Admin Panel → Course list → Edit** — navigates to `#/classes/<id>` which loads the course builder
3. **Course Builder Page** (admin view):
   - **Left palette**: Sections (overview, syllabus, schedule, speakers, files, simulations, lectures, homeworks, exams)
   - **Right palette**: Widgets (text block, image, video, link list, embed, divider)
   - **Center canvas**: Drop zone where dragged sections/widgets stack vertically
   - Each dropped item becomes an editable block in the canvas
   - Blocks can be reordered by dragging within the canvas, or removed via × button
   - **Autosave**: dirty flag set on any edit; timer fires every 30s if dirty; saves to Firestore; status indicator shows "Saved" / "Saving..." / "Unsaved changes"
4. **Student/Speaker view**: Renders the saved layout read-only (no palettes, no drag handles)

## Approach

### Phase 1: Autosave Infrastructure
- Add `_dirty` flag and `_autosaveTimer` to module state
- `markDirty()` sets flag + updates status indicator
- `startAutosave()` sets 30s interval; if dirty, calls `persistAll()` which saves the full schedule doc
- `persistAll()` gathers current canvas state → writes to Firestore → clears dirty flag → shows "Saved"
- Wire autosave start in `wireClassPage()`, clear on page leave (hashchange listener)

### Phase 2: Canvas Layout
- Replace `buildClassLayout()` sidebar+content with a three-column layout:
  - Left column: section palette (draggable items)
  - Center column: canvas (drop zone, stacked blocks)
  - Right column: widget palette (draggable items)
- Student/speaker view renders center column only (read-only blocks, no palettes)
- Store layout as ordered array in schedule doc: `layout: [{ type: 'section', key: 'overview' }, { type: 'widget', kind: 'text', content: '...' }, ...]`

### Phase 3: Drag and Drop
- Use HTML5 Drag and Drop API (native, no library needed)
  - Palette items: `draggable="true"`, `dragstart` sets transfer data
  - Canvas: `dragover` (prevent default + show drop indicator), `drop` (insert block)
  - Reorder: canvas blocks also draggable; `dragover` on blocks shows insertion line between them
- Drop indicator: thin colored line between blocks showing where the item will land
- Each block in canvas gets a drag handle (⠿) and remove button (×)

### Phase 4: Block Renderers
- **Section blocks** reuse existing component renderers (text editor, file manager, speakers scheduler) but inline in the canvas rather than in a separate content area
- **Widget blocks** are new, lightweight content types:
  - `text`: Rich-ish textarea (same as current text component)
  - `image`: Single image upload + caption
  - `video`: Video embed (URL or upload)
  - `link-list`: Ordered list of labeled URLs
  - `embed`: Raw HTML/iframe embed (admin only)
  - `divider`: Visual separator (hr)

### Phase 5: Persistence Schema
- `schedules` doc gains a `layout` array field:
  ```
  layout: [
    { type: 'section', key: 'overview', id: 'blk_1' },
    { type: 'widget', kind: 'text', id: 'blk_2', content: 'Welcome to...' },
    { type: 'section', key: 'speakers', id: 'blk_3' },
    { type: 'widget', kind: 'divider', id: 'blk_4' },
    { type: 'section', key: 'files', id: 'blk_5' }
  ]
  ```
- Section blocks reference `SECTION_REG` key; content stored in existing fields (`overviewContent`, etc.)
- Widget blocks store content inline in the layout array entry
- `sections` array field kept for backwards compatibility (derived from layout on save)
- Block IDs are short random strings (`blk_` + 6 hex chars) for stable DOM keys

| File | Changes |
|------|---------|
| `class-scheduler.js` | Replace sidebar layout with canvas layout; add drag-and-drop handlers; add autosave; add widget renderers; add block management (add/remove/reorder) |

| `storage.rules` | No changes needed (classes path already writable) |

## Key Decisions

- **HTML5 DnD over library**: Keeps the no-dependencies philosophy. Touch support via `touchstart`/`touchmove`/`touchend` polyfill if needed (or `pointer events` like the scheduler's freeform drag).
- **Layout array in schedule doc**: Single source of truth for page structure. Simpler than separate collection for blocks.
- **Widgets vs sections**: Sections are the existing heavyweight components (text editor, file manager, speakers). Widgets are lightweight inline content blocks. Both live in the same layout array.
- **Autosave over manual save**: Better UX for a builder page. 30s interval is a good balance between responsiveness and Firestore write costs.
- **Backwards compatibility**: If `layout` field is missing, fall back to rendering `sections` array in the old sidebar style. Migration is implicit — first admin edit creates the layout array.

## Risks

- **Drag and drop on mobile**: HTML5 DnD has poor mobile support. May need a touch polyfill or long-press-to-drag pattern. Could defer mobile editing to Phase 2 and show read-only on mobile for now.
- **Speakers section complexity**: The scheduler component expects a dedicated container. Embedding it inline in a canvas block may need careful container ID management.
- **Autosave conflicts**: If two admins edit simultaneously, last-write-wins. Acceptable for a lab website.
