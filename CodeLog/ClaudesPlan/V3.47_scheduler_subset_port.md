# V3.47 — Scheduler: subset native port (My Schedulers tab)

## Goal

Native RM port of the **My Schedulers** half of `/apps/scheduler/`.
Create + manage shareable scheduler links (lab seminar lineup, time
poll, review panel) move into [rm/pages/scheduler.html](../../rm/pages/scheduler.html);
the editor view continues to mount the parent `/scheduler.js` engine
(1,382 LOC, stateless) so the calendar / sessions builder / freeform
grid / admin-guest-public view switcher all keep working unchanged.

The **My Schedule** tab (personal calendar layers, recurring
availability, Google Calendar OAuth) is deferred to V3.51 alongside
the equipment OAuth refactor. The standalone `/apps/scheduler/` URL
still resolves until then; users who need the deferred features can
hit it directly. The Phase A iframe wrapper at
[rm/pages/app-scheduler.html](../../rm/pages/app-scheduler.html) is
replaced with a `<meta refresh>` redirect so bookmarks land in the
right place.

## Architecture

The lab app's `/apps/scheduler/app.js` mixes three concerns:

1. **Scheduling engine** — already abstracted into `/scheduler.js`
   (renderable HTML + wirable events, zero Firestore). The engine is
   loaded by both the public-site `#/dashboard/scheduler` editor and
   the lab app today, so it's a stable contract. The RM page loads it
   the same way: `<script src="/scheduler.js">`.
2. **My Schedulers UI** — list view + create form + editor mount
   (~250 LOC of the 1,174-LOC app.js). This is what V3.47 ports.
3. **My Schedule UI + ScheduleService + CalendarService** — the
   personal calendar layer system. **Not ported** in V3.47.

The new [rm/js/scheduler.js](../../rm/js/scheduler.js) (462 LOC) does
three things:

- **List view** (own + admin-sees-all): `api.load('scheduler/list.json')`
  → client-side filter by `ownerUid` → render the list with `+ New
  Scheduler` button, `Manage` / `Delete` actions. `LIVE_SYNC.attach`
  on `scheduler/list.json` so cross-tab + multi-user creates show up.
- **Create flow**: surgical `db().collection('schedules').doc(id).set(...)`
  via firebridge.db(). On success, `loadAndRender()` to refresh the
  list.
- **Editor view**: lifts the entire editor mount from app.js lines
  1088-1172 — load schedule + speakers via direct `firestore`
  reads, build the engine `config` object with `onSaveSchedule` /
  `onSaveSpeaker` / etc. callbacks routing to surgical writes, call
  `McgheeLab.Scheduler.render(config)` + `wire(containerId, config)`.

## Engine API (from /scheduler.js)

The engine is **stateless** and emits HTML + wires events. The host
supplies persistence callbacks. Public surface attached to
`window.McgheeLab.Scheduler`:

| Method | Returns | Notes |
|---|---|---|
| `render(config)` | HTML string | renders the active view (admin/guest/public) |
| `wire(containerId, config)` | void | attaches event listeners after render |
| `renderGrid(config)` | HTML string | availability grid (sessions or freeform) |
| `renderSetupForm(schedule)` | HTML string | full schedule builder |
| `wireSetupForm(config)` | void | builder mode toggle, save, etc. |
| `renderBuilder(...)` | HTML string | calendar + time-grid builder |
| `wireBuilder(...)` | API obj | `{getBlocks, getCells, getDays, getDuration}` |
| `expandSchedule(schedule)` | `{days, slots, allSlots\|blocks}` | derive display state |
| `optimizeSchedule(speakers)` | array | greedy slot assignment |
| `renderConfirmedSpeakers(...)` | HTML string | confirmed speaker list |
| `heatColor(count, max)` | CSS color | heatmap cells |
| `fmtTime(s)` / `slotLabel(...)` | string | utilities |

Config shape consumed by `render` + `wire`:

```js
{
  scheduleId, schedule, speakers,
  currentSpeaker, viewType, useKeyAuth,
  adminViewMode, previewSpeakerIdx,
  buildInviteURL: (sid, key) => string,
  onSaveSpeaker:   async (id, data) => void,
  onSaveSchedule:  async (data) => void,
  onAddSpeaker:    async (data) => void,
  onDeleteSpeaker: async (id) => void,
  onRefresh:       async () => void,
  onSwitchView:    (mode, idx) => void,
}
```

## Data layer

Two existing Firestore collections — both already have
firestore.rules in place from earlier; **no rules changes needed**:

- **`schedules/{id}`** — public read; create=any auth; update/delete =
  owner OR admin. Rule at [firestore.rules:189](../../firestore.rules#L189).
  Fields: `id`, `title`, `subtitle`, `description`, `mode`
  (`'sessions'|'freeform'`), `sessionBlocks[]`, `selectedDays[]`,
  `startDate`, `endDate`, `slotDefs[]`, `freeformCells[]`, `startHour`,
  `endHour`, `granularity`, `sessionDuration`, `guestFields[]`,
  `guestInstructions`, `booleanQuestions[]`, `sections[]`, `ownerUid`,
  `updatedAt`.
- **`participants/{key}`** — public read; create = admin OR auth + key
  match (the invite-key flow). Rule at
  [firestore.rules:199](../../firestore.rules#L199). Docs are keyed on
  the invite-key UUID, not Firebase auto-id. Fields: `key`,
  `scheduleId`, `speakerName`, `speakerEmail`, `speakerUid`,
  `availability[]`, `assignedSlot`, `talkSummary`, `questions[]`,
  `presentationLink`, `booleanAnswers{}`, `role`, `createdAt`,
  `updatedAt`.

The list view goes through `api.load` for cache + LIVE_SYNC
(re-fetch on snapshot). The editor view does direct one-shot reads
because it's single-editor-at-a-time and the engine's `onRefresh`
callback already handles in-session reloads.

New api-routes.js entry: `scheduler/list.json` →
`collection: 'schedules', wrapKey: 'schedules', cache: MEDIUM`.
Lab-scope, no shadowJson — schedules has been Firestore-only from
day one. The guest invite link still works (`#/schedule/{id}?key=...`)
because public read on both collections satisfies the unauthenticated
flow.

## Key decisions

- **Engine stays at /scheduler.js.** It's already used by the public
  site's `#/dashboard/scheduler` editor; duplicating it into RM would
  create drift. Same-origin loads from the RM page work fine.
- **Subset scope.** Doing the full port (My Schedule + CalendarService)
  in V3.47 would either bring the OAuth refactor forward (V3.51 was
  scoped that way deliberately) or duplicate ~1,500 LOC of shared-
  service code. Subset keeps the change small and reversible.
- **CSS lift, not link.** Linking the lab `styles.css` directly would
  break when V3.51 deletes `/apps/scheduler/`. Copying the file with
  a var-alias header is the same pattern V3.41 meetings used.
- **No firestore.rules changes.** Schedules + participants rules
  predate the migration and already permit the V3.47 access patterns
  (admin or owner manages, public read for guest invites). No deploy
  needed.
- **No LIVE_SYNC on the editor.** Single-editor-per-schedule in
  practice; `onRefresh` callback handles in-session reloads. The
  list view does subscribe so cross-tab creates / deletes update.

Smoke flow:

1. Sign in as lab member → `My Schedulers` list. Click `+ New Scheduler`,
   create one with title + sessions mode → list updates.
2. Click `Manage` → editor mounts. Engine renders the calendar /
   sessions builder. Drag a time window, confirm the session appears.
3. Hit `Save` (engine button) → `onSaveSchedule` callback fires →
   surgical write to `schedules/{id}`. Reload page → schedule shape
   persists.
4. Add a speaker via the engine's UI → `onAddSpeaker` fires → new doc
   in `participants/{key}`. Copy invite URL, open it in another
   browser tab → public-site guest view loads (served by the SPA at
   `#/schedule/{id}?key=...`).
5. Click `← All Schedulers` → back to list view.
6. `http://localhost:8000/rm/pages/app-scheduler.html` → meta-refreshes
   to `/rm/pages/scheduler.html`.
7. `http://localhost:8000/apps/scheduler/` → standalone fallback still
   works (My Schedule tab accessible there until V3.51).
8. DevTools → Application → IndexedDB → `rm-cache` shows
   `route::scheduler/list.json` after first load.
