# V3.56 — Split View: side-by-side RM pages

## Goal

Add a general-purpose split-view feature to RM. User opens any two
RM pages side-by-side in a draggable two-pane shell. Explicitly
flagged on the user's V3.40 follow-up list:

> "we really like the ability to have multiple pages open in a split
>  view. this could be benefitial for RM in general"

The lab-apps.js `_splitAppId` / `_splitRatio` harness preserved
through V3.53 cleanup is the reference; this rebuilds the same
pattern as a clean RM-native shell.

## Architecture

**Iframe-based shell at a dedicated page.** Cleaner than wiring
conditional split-rendering into every RM page:

- New page: [rm/pages/split-view.html](../../rm/pages/split-view.html).
- New renderer: [rm/js/split-view.js](../../rm/js/split-view.js).
- New stylesheet: [rm/css/split-view.css](../../rm/css/split-view.css).
- Each "regular" RM page stays untouched. The split shell loads each
  page in its own iframe, reusing the existing native renderer
  end-to-end.

**Same-origin auth share.** Iframes loaded from `/rm/pages/<name>.html`
share Firebase IndexedDB persistence with the parent. Each iframe's
own `firebase.auth()` resolves to the same signed-in user
automatically — no postMessage handshake needed (unlike the V3.40
iframe-bridge era which had to forward auth via `mcgheelab-auth`
postMessages because the lab apps lived under `/apps/` rather than
RM's own scope).

**URL-driven state.** Single source of truth is the query string:

```
/rm/pages/split-view.html?left=/rm/pages/X.html&right=/rm/pages/Y.html&ratio=0.55
```

Bookmarkable. Survives page reload. `history.replaceState` writes
back on every dropdown change / divider drag so the URL always
reflects the current split.

## What lands

### Page shell

Two iframes side-by-side under a draggable 6px divider. Each pane
has a small header bar (~32px) above the iframe with:

| Control | Behavior |
|---|---|
| **Page dropdown** | Lists every page in `nav.js`'s `NAV_ITEMS` (children flattened). Selecting one swaps just THAT pane's iframe — the other pane keeps its scroll, in-progress edits, etc. The Split View page itself is excluded from the dropdown to prevent confusing recursion. |
| **Reload (↻)** | Reloads only this pane's iframe. |
| **Open in full window (⤢)** | Navigates the parent window to this pane's URL — i.e. exits split view by promoting this pane to the full window. |
| **Close (×)** | Exits split view; navigates the parent window to the OTHER pane's URL (the "survivor"). |

### Drag-to-resize

`pointerdown` on the divider sets `dragging = true` and adds
`body.split-dragging` (which adds `pointer-events: none` to the
iframes so the drag isn't captured by inner pointer handlers).
`pointermove` updates the `_ratio` clamped to `[0.15, 0.85]` and
re-applies via `flex-basis` on each pane (no full re-render).
`pointerup` writes the final ratio to the URL.

### Mobile (≤700px viewport)

Stacks vertically. Divider becomes horizontal with `cursor:
row-resize`. Page-swap dropdowns truncate to 140px max-width to
fit the narrower header. Mobile is a fallback layout, not a
primary use case for split view (most students will use it on
laptop+).

### Nav entry

`{ label: '⊟ Split', href: '/rm/pages/split-view.html' }` added as
the last top-level entry in `NAV_ITEMS`. The leading `⊟` glyph
distinguishes it visually from the regular section dropdowns. No
children — clicking opens the split shell directly with default
state (left = right = `/rm/index.html`, ratio = 0.5).

## Defaults

When no query params are present, the shell loads with:

- **left**: `/rm/index.html` (Dashboard)
- **right**: `/rm/index.html` (Dashboard — same on both sides)
- **ratio**: 0.5

The user picks what to load via the per-pane dropdowns. We
deliberately don't try to be "smart" about default content (e.g.
"if user came from page X, default left to X") — keeping the shell
stateless and predictable matters more than guessing.

**No firestore.rules / functions changes.** The split shell is pure
DOM glue — no Firestore reads, no Cloud Function calls. The pages
loaded inside the iframes touch Firestore the same way they would
when loaded directly.

## Trade-offs

- **Iframes load duplicate code.** Each iframe re-loads the full RM
  bundle (firebase SDK + firebase-bridge + adapter + nav etc., ~500KB
  per iframe). For two iframes that's roughly 1MB total instead of
  500KB for a single page. Not great, but acceptable for an explicit
  "split view" mode the user opts into. A future V3.57+ enhancement
  could convert to direct-mount (each page's renderer becomes a
  function that mounts into a container) to eliminate the duplication
  — but that requires touching every RM page renderer.
- **Two top-nav bars when nested.** Each iframe shows its own RM
  top nav inside the pane. So the user sees: parent top nav + pane
  header + iframe top nav + pane content. Functional but visually
  busy. A future enhancement could pass a `?embed=1` query param to
  the iframe and have nav.js skip rendering when present.
- **Iframe pointer events suppressed during divider drag.** Without
  this, the iframe's own pointer handlers capture the drag and the
  divider freezes mid-resize. Trade-off is that you can't keep
  interacting with iframe content while resizing.

Smoke flow:

1. Sign in as lab member. Click `⊟ Split` in the RM top nav. Page
   loads with two Dashboard iframes side-by-side at 50/50.
2. Use the left pane's dropdown → switch to "Activity Tracker" →
   only the left pane reloads.
3. Use the right pane's dropdown → switch to "Lab Chat" → right
   pane loads chat.
4. Drag the divider → both panes resize live; releasing writes the
   ratio to the URL (visible in the address bar).
5. Reload the page → previous left/right/ratio restored from the
   URL.
6. Click the right pane's "Open in full window" → parent navigates
   to chat.html (split view exited, chat is the active page).
7. Hit back → returns to the split view (URL is in browser history).
8. Click the left pane's "×" close → parent navigates to the
   right-pane URL (chat).
9. Mobile (resize viewport <700px) → panes stack vertically,
   divider becomes horizontal.
10. DevTools → Application → Cache Storage shows
    `mcgheelab-shell-v31`.
