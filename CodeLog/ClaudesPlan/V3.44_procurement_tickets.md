# V3.44 — Procurement: unified ticket lifecycle page

## Goal

Build a single new RM page that covers the whole purchase pipeline:
**request → approve → order (PO upload) → receive (mark from open-orders
list, optional receipt upload) → place (record location) → inventory
(auto-create the item)**. One page, one collection, one continuous
audit trail per ticket.

This supersedes the disconnected pieces today:

| Today | Purpose | Phased out |
|---|---|---|
| `/apps/purchases/` (lab app) | Student request submission → `purchaseRequests` | V3.45 (after burn-in of new page) |
| `/apps/procurement/` (lab app) | Student receipt submission → `procurement` | V3.45 |
| [rm/pages/purchase-requests.html](../../rm/pages/purchase-requests.html) | Admin review of `purchaseRequests` | Kept during burn-in |
| [rm/pages/receipts.html](../../rm/pages/receipts.html) | Admin manual orders + parsed receipts + budget analysis on `financeReceipts` | Renamed nav-side to "Receipts & Budget" — orthogonal feature |

## Data model

New Firestore collection: **`procurementTickets/{ticketId}`**. Single
document carries every lifecycle stage; the `status` field gates which
fields and which UI affordances apply.

```
{
  id, requestedBy, requesterName,
  itemDescription, vendor, catalogueNumber,
  estimatedCost, quantity,
  project, fundingAccount,
  reason,                           // NEW — "why do you need this?"
  justification,                    // legacy free-text (purchaseRequests parity)
  urgency: 'routine' | 'needed_soon' | 'urgent',
  category, // consumable|reagent|equipment|software|office|other

  status: 'requested' | 'approved' | 'denied' | 'ordered' | 'received' | 'placed',

  piNotes, approvedBy, approvedByName, approvedAt,

  poNumber, poUrl, poStoragePath,
  orderDate, expectedDelivery,

  receivedAt, receivedBy, receivedByName,
  receiptUrl, receiptStoragePath, actualCost,

  location, placedAt, placedBy, placedByName, inventoryItemId,

  createdAt, updatedAt
}
```

On placement, an inventory record is created at `inventory/{newId}`
with `kind: 'item'`, `procurementTicketId: ticket.id`, the captured
location, quantity, and unit price. The ticket's `inventoryItemId` is
stamped with the new doc id so the two stay linked.

## UI

Single page at [rm/pages/procurement.html](../../rm/pages/procurement.html).
Tabs across the top, gated by role:

| Tab | Visible to | Filter |
|---|---|---|
| `+ Submit Request` | everyone | (form) |
| `My Tickets` | everyone | `requestedBy === uid` |
| `Pending` | admin | `status === 'requested'` |
| `Awaiting Order` | admin | `status === 'approved' && !poUrl` |
| `Open Orders` | everyone | `status === 'ordered'` |
| `Awaiting Placement` | everyone | `status === 'received'` |
| `Archive` | everyone | `status in ['placed', 'denied']` (with search) |

Each ticket renders as a card showing: title + chips (status, urgency),
meta row (requester, project, funding, est. cost, qty, cat#),
why/justification, and a stack of stage-stamps (✅ approved, 📄 PO,
🚚 ETA, 📦 received, 📍 placed, 📚 in-inventory). The action bar at the
bottom shows only the buttons that apply to the current stage:

- `requested`: Approve / Deny (admin); Cancel (own)
- `approved`: Upload PO & mark Ordered (admin) — modal with PO #, order
  date, ETA, file upload (PDF or image)
- `ordered`: Mark Received (anyone) — modal with actual cost + optional
  receipt upload
- `received`: Record Placement & add to Inventory (anyone) — modal with
  location (required), quantity, unit price, category, notes
- `placed`: Open in Inventory (link)

## Deploy steps

After commit:

```bash
firebase deploy --only firestore:rules
firebase deploy --only storage:rules
```

The Firestore rule grants `isLabMember()` access to a brand-new
collection — without the rules deploy, every read/write returns
permission-denied and the page surfaces "Failed to load procurement —
see console." Same for Storage uploads.

## Key decisions

- **New collection, not extending `purchaseRequests`.** Old
  `purchaseRequests` stays alive for read-only burn-in; the existing
  [rm/js/purchase-requests.js](../../rm/js/purchase-requests.js)
  admin-review page keeps working unchanged. After V3.45 retires the
  lab apps, a future phase can sweep the old collection (data
  migration) once the new page proves out.
- **One ticket, one document.** A status field with 6 values is
  simpler than splitting the lifecycle across `purchaseRequests` →
  `purchaseOrders` → `receivedItems` collections. The card UI
  collapses the historical stamps into a single timeline; queries
  filter on `status`.
- **Permissive update rule.** Locking each field to a specific role
  in firestore.rules would either (a) need 6 separate rule blocks
  per status transition, or (b) require dedicated subcollections.
  Both are heavy for a small lab where trust is high. UI gating +
  audit fields (approvedBy / receivedBy / placedBy + updatedAt) are
  the right enforcement layer here.
- **Storage path unified at `procurement/{docId}/**`.** Reusing the
  existing rule keeps the deploy minimal. Both PDFs (typical for
  PO/iLab invoice) and images (typical for phone-snapped receipts)
  are accepted.
- **Inventory hand-off lives in this page.** Doing the
  `inventory/{id}` create here (rather than a separate "promote to
  inventory" step) means the placement modal collects exactly the
  fields inventory needs: location, qty, unit price, category. The
  inventory page already loads via [rm/js/inventory.js](../../rm/js/inventory.js)
  with `kind: 'item'` rows, so the new doc shows up immediately.
- **Surgical writes + LIVE_SYNC, mirroring V3.41 meetings pattern.**
  `_live = LIVE_SYNC.attach({...})` on boot; every write sets
  `_live.suppressUntil = Date.now() + 2500` and then mutates the
  in-memory list + re-renders so the user sees their action
  instantly without waiting for the suppress window to close.
- **Lab apps `/apps/purchases/` and `/apps/procurement/` stay alive
  for V3.44.** Cutover risk is real — students may have ticket
  drafts open. V3.45 deletes them once V3.44 is verified in
  production for a few days.

Smoke flow (single-user end-to-end):

1. Sign in as admin → `+ Submit Request` form. Fill description,
   reason, project, urgency. Submit.
2. Switch to `My Tickets` → see the new card with status `requested`.
3. Switch to `Pending` (admin tab) → see the same ticket. Click
   `Approve` → enter PI note → status becomes `approved`.
4. Switch to `Awaiting Order` → click `Upload PO & mark Ordered`.
   Enter PO number, attach a small PDF, set order date + ETA,
   confirm. Status → `ordered`. The PO link should resolve to a
   downloadable file in Firebase Storage.
5. Switch to `Open Orders` → click `Mark Received`. Optional: attach
   a receipt photo. Confirm. Status → `received`.
6. Switch to `Awaiting Placement` → click `Record Placement & add
   to Inventory`. Enter location like "Cold room shelf 3", confirm.
   Status → `placed`. Toast shows "Marked placed & added to
   inventory".
7. Click `Open in Inventory` → [rm/pages/inventory.html](../../rm/pages/inventory.html)
   loads; the new item appears with name, vendor, location filled.
8. Open the same procurement URL in a second tab → mutate in tab 1
   → tab 2 re-renders within ~1s after the suppress window closes
   (LIVE_SYNC fan-out).

Negative checks:

- Sign in as guest → pending-access overlay covers the page
  (`firebridge.gateSignedIn`).
- Sign in as non-admin lab member → `+ Submit Request`, `My
  Tickets`, `Open Orders`, `Awaiting Placement`, `Archive` tabs
  visible; `Pending` and `Awaiting Order` (admin tabs) hidden.
- Try to delete a ticket as non-admin → button not rendered;
  Firestore rules would refuse anyway.
