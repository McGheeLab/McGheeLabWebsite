# V3.32 — Auth Race Condition Fix (First-Time Login)

## Goal
Permanently fix the recurring race condition where first-time or cold-start logins to lab apps show the "Sign in required" auth wall despite the user being authenticated. Also fix the related bug where login itself times out after refreshing during a stuck-loading state.

## Problem Analysis
Five interconnected issues contribute to the failure chain:

1. **Standalone mode treats initial `null` as definitive** — `onAuthStateChanged` can fire with `null` before Firebase persistence finishes restoring the session from IndexedDB. The bridge calls `_fireAuthFailed()` immediately, replacing the app DOM with an auth wall.

2. **No recovery from `_fireAuthFailed()`** — Once the auth wall is shown and `_ready = true`, a later `onAuthStateChanged` with the real user calls `_fireReady()` on destroyed DOM. The app never re-renders.

3. **Embedded mode is timing-based, not event-driven** — The parent sends auth retries at load+0/200/600/1500ms then stops. If the parent's own Firebase auth hasn't resolved during that window, every retry sends `user: null`. No mechanism re-sends auth when the parent's auth state later resolves.

4. **`Auth._authStateResolved` never implemented** — `lab-apps.js` references `Auth._authStateResolved` to decide whether to show "Loading..." vs redirect to login, but the property was never added to the Auth object. The check always evaluates to `undefined === false` → `false`, making it dead code.

5. **Service worker caches `accounts.google.com`** — Google Auth endpoints are cached with stale-while-revalidate. After clearing browser data and reloading, the SW can serve stale/corrupt Google Identity responses, causing login to hang. Additionally, login calls have no timeout, so a hung request blocks the UI forever.

6. **Auth.onChange listener accumulation** — `wireLabApp()` adds a new `Auth.onChange` listener each time it's called (app switching, split-view), but never removes old ones. Stale listeners reference old iframe elements.

## Approach

### auth-bridge.js
- **Grace period for null**: In standalone mode, when `onAuthStateChanged` fires with `null`, start a 2s `_nullSettleTimer`. If a real user arrives during that window, cancel the timer. If it expires, show the auth wall.
- **Recovery path**: New `_authFailed` flag. If `_fireReady()` is called with a valid user while the wall is showing, trigger `location.reload()`.
- **Embedded null guard**: Don't call `_fireAuthFailed()` when parent sends `user: null` — rely on the 8s timeout.
- **`onReady()` guard**: Don't fire callback with null when auth wall is showing.

### user-system.js
- **`Auth._authStateResolved`**: Add the property, set to `true` inside `onAuthStateChanged`. Makes the `lab-apps.js` loading-screen logic functional.
- **`Auth.offChange(fn)`**: Add unsubscribe method for listener cleanup.
- **Login timeouts**: Wrap `signInWithEmailAndPassword` and `signInWithPopup` in `Promise.race` with a 15s timeout that surfaces a user-friendly error message.

### lab-apps.js
- **Event-driven auth forwarding**: Register `Auth.onChange` listener that re-sends to iframes.
- **Listener cleanup**: Store listener references in module-level vars. On each `wireLabApp()` call, remove old listeners before adding new ones.
- **Message handler consolidation**: Single `window.message` listener that sends to whichever iframes are active, instead of per-iframe listeners.

### sw.js
- **Remove `accounts.google.com` from CDN cache rules**: Move it to an explicit network-only exclusion alongside `apis.google.com`, above the CDN stale-while-revalidate block.

## Files Modified
- `apps/shared/auth-bridge.js` — standalone grace period, recovery reload, embedded null guard
- `user-system.js` — `_authStateResolved`, `offChange()`, login timeouts
- `lab-apps.js` — Auth.onChange forwarding with cleanup, consolidated message handler
- `sw.js` — Google auth endpoints excluded from caching

## Key Decisions
- Recovery uses `location.reload()` rather than trying to re-init the app in-place. The DOM is already destroyed by the auth wall, and each app's init logic is complex. A clean reload is reliable and simple.
- Grace period is 2s — long enough for persistence to resolve, short enough that genuinely logged-out users don't wait long.
- Login timeout is 15s — generous for slow connections but prevents infinite hangs.
- Embedded timeout stays at 8s. The parent-side Auth.onChange listener makes this mostly a safety net now.
