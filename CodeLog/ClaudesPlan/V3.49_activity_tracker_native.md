# V3.49 — Activity Tracker: native RM port

## Goal

Replace the V3.40 iframe wrapper at [rm/pages/activity-tracker.html](../../rm/pages/activity-tracker.html)
with a native renderer. The lab app at `/apps/activity-tracker/`
(2,100 LOC + 858 LOC CSS) was the most-used app pre-migration; this
port brings it onto the RM contract (firebridge auth gate, IndexedDB
cache via the api adapter for ancillary loads, no MobileShell, no
postMessage iframe-resize).

Calendar Integration is intentionally not loaded (CalendarService
Google OAuth is V3.51 scope) — the renderer's calendar-suggest panel
calls `McgheeLab.CalendarService?.getEventsForDate(...)` defensively;
without the service loaded, the panel just stays empty. Everything
else — daily logging, hierarchical category tree (synced from RM via
rm-categories.js), client-side ML categorization (Naive Bayes),
Anthropic API categorization, Web Speech voice input, Huddle plan
integration, weekly analytics with Chart.js, AI insights — all ports
verbatim.

## Approach

Followed the V3.41 meetings playbook: copy the lab app's `app.js`
into `rm/js/activity-tracker.js`, then surgically patch the boot
section + container id + db() helper. The renderer's 1,900+ LOC of
view code, ML model, Anthropic API calls, Chart.js setup, and
Firestore operations transfer verbatim because the data shape is
identical (user-scope `trackerData/{uid}` and `trackerEntries/{uid}/entries/{entryId}`).

### Patches applied to the lifted file

1. **`window.McgheeLab = window.McgheeLab || {}`** at the top — the
   MobileShell / CalendarService / RmCategories optional-chaining
   lookups need the namespace to exist (otherwise
   `McgheeLab.MobileShell?` is a `ReferenceError`, not `undefined`).
   `rm-categories.js` also asserts this on its own line 47, but
   re-asserting here keeps the renderer self-bootstrapping if loaded
   alone.
2. **Container id `#app` → `#tracker-root`** — avoids collisions with
   other RM pages that may use `#app`.
3. **`db()` prefers `firebridge.db()`** when available, falls back to
   `firebase.firestore()`.
4. **Bootstrap rewrite** — replaces the dual AppBridge / direct-Firebase
   path (lines 120–178) with the RM contract:
   `firebridge.gateSignedIn()` → `await firebridge.whenAuthResolved()`
   → `_user = firebridge.getUser()`, `_profile = firebridge.getProfile()`
   → `loadData()` → `render()`. Mobile tab swipe (`MobileShell.enableTabSwipe`)
   stays guarded by `if (McgheeLab.MobileShell && ...)` so it no-ops in
   RM. Profile-role flips re-render via `firebridge.onAuth`.
5. **No LIVE_SYNC.attach.** Trackerdata and trackerEntries are
   user-scope and single-user-edit in practice; a snapshot listener
   would only echo our own writes. Re-render after each surgical
   write keeps the UI fresh without the round-trip.

The remaining ~2,000 LOC of view + ML + AI + Chart code transfers
unchanged.

## Deferred to V3.51

- **Calendar Integration import** — the calendar-suggest panel pulls
  events via `McgheeLab.CalendarService?.getEventsForDate(date)`. With
  `CalendarService` not loaded, the optional chaining returns
  `undefined` and the panel stays empty. V3.51's OAuth refactor moves
  Google/Outlook tokens into `oauthConnections/{uid}` Firestore docs
  (instead of `sessionStorage`); after that lands, RM can load
  `CalendarService` and the import works again.
- **Sunset `/apps/activity-tracker/`** — the standalone URL still
  resolves until V3.51 deletes the directory. Users who need
  Calendar import in the meantime can hit the standalone URL.

## Key decisions

- **Verbatim lift, not rewrite.** 2,000 LOC of view code is correct
  and battle-tested; mechanically transcribing it onto a "different"
  RM-idiom renderer would introduce regressions. Same call I made
  for V3.41 meetings; same results.
- **rm-categories.js copied into RM.** Phase C deletes `/apps/shared/`
  outright. The RM copy is the canonical version going forward; if
  the public-site SPA's #/dashboard editors still need the bridge
  later (they shouldn't — those edit `labConfig/categorySchema`
  directly), they get their own copy.
- **Chart.js via CDN.** The renderer's analytics dashboard expects
  `Chart` as a global; CDN-loaded UMD bundle satisfies that without
  changing renderer code.
- **No LIVE_SYNC.attach.** User-scope, single-user-edit; LIVE_SYNC
  costs more than it saves. The data layer's surgical writes drive
  re-renders directly.
- **Calendar deferred, not stubbed.** Easier to leave the optional
  chaining and let CalendarService load no-op'd than to introduce a
  mock. V3.51 simply needs to script-load `CalendarService` from
  `/rm/js/oauth-connections.js` after the OAuth refactor.

Smoke flow:

1. Sign in as guest → pending-access overlay covers the page.
2. Sign in as lab member → page renders with sidebar (Daily / Weekly
   / Analytics / Categories / Settings) and the Daily view active.
3. Type a task in the input + tab to add → entry persists. Reload →
   row still there.
4. Switch to Weekly → 7-day grid renders.
5. Switch to Analytics → Chart.js renders distribution + trend +
   category + milestones charts.
6. Switch to Categories → tree view of `labConfig/categorySchema`
   (read-only — categories live in RM's labConfig admin).
7. Switch to Settings → AI key status (sourced from `users/{uid}.anthropicKey`),
   ML model stats, working hours, export JSON.
8. Open the same URL in a second tab → mutate in tab 1 → tab 2's
   data is stale until next reload (no LIVE_SYNC, by design).
9. Calendar-suggest panel on Daily view stays empty (Calendar
   integration deferred).
10. `/apps/activity-tracker/` standalone URL still loads as a fallback.
11. DevTools → Application → Cache Storage → only `mcgheelab-*-v24`
    caches.
