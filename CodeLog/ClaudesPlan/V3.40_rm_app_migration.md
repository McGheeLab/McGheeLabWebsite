# V3.40 — RM app migration: Phase A (iframe bridges + nav restructure)

## Goal

Begin the migration of the public-site `/apps/` lab apps and admin/dashboard surface into RM at `/rm/`. After this phase, RM becomes the single nav home for every lab tool; each app is reachable via an iframe-wrapped RM page until it gets ported natively in V3.41–V3.52. The public site loses its **Lab Apps** menu — the conditional **RM** link in the public nav becomes the sole gateway for logged-in lab members.

This is the first of three phases. **Phase B** (V3.41–V3.52) ports each app onto the RM data contract (`api.load`/`api.save`, IndexedDB cache, `LIVE_SYNC`, `firebridge` auth gate). **Phase C** (V3.53) deletes `/apps/`, the public-site dashboard/admin/cv/guide routes, and the legacy `lab-apps.js`/`cv-builder.js`/`scheduler.js`/`class-builder.js` loaders.

## Approach

### Six-group RM nav

Replace the current nine-item top nav in [rm/js/nav.js](../../rm/js/nav.js) with a tighter six-group layout: **Dashboard | Activity | Research | Operations | Lab Admin | Settings**. The existing per-item `gate: 'admin' | 'lab-member'` machinery + auto-collapse-empty-group logic at lines 163-194 carries over unchanged.

- **Operations** (gate: `lab-member`): chat, huddle, meetings, scheduler, equipment — every link points at `/rm/pages/app-<name>.html` iframe wrappers.
- **Lab Admin** (gate: `lab-member`): compliance, chemicals, inventory, people, important-people, career, receipts (procurement), purchase-requests, projects-grants, budget, analytics, finance — these all hit existing native RM pages.
- **Settings** (gate: `lab-member`): profile, settings, cv-overview, cv-editor, member-activity (gate: `admin`).
- **Activity** keeps its current shape (Tracker/Overview/Calendar/Email/Tasks/Sharing/Year). The Tracker entry still points at the existing iframe wrapper at [rm/pages/activity-tracker.html](../../rm/pages/activity-tracker.html); it's not rewritten until V3.49.
- **Research** absorbs Teaching and Service as additional submenu entries to keep the top nav at six items rather than eight.

### Iframe bridges

Five new wrappers under [rm/pages/](../../rm/pages/), all parameterized copies of [rm/pages/activity-tracker.html](../../rm/pages/activity-tracker.html) — only `<title>`, `IFRAME_SRC`, and the loading-card text differ:

- [rm/pages/app-chat.html](../../rm/pages/app-chat.html) → `/apps/chat/`
- [rm/pages/app-equipment.html](../../rm/pages/app-equipment.html) → `/apps/equipment/`
- [rm/pages/app-huddle.html](../../rm/pages/app-huddle.html) → `/apps/huddle/`
- [rm/pages/app-meetings.html](../../rm/pages/app-meetings.html) → `/apps/meetings/`
- [rm/pages/app-scheduler.html](../../rm/pages/app-scheduler.html) → `/apps/scheduler/`

The wrapper preserves the proven postMessage handshake (`mcgheelab-app-ready` from the iframe → `mcgheelab-auth` payload back). Same-origin Firebase IndexedDB persistence means the iframe's own `firebase.auth()` already has the user; the postMessage just populates `auth-bridge.js`'s `_user`/`_profile` for code paths that read those directly. The null-profile guard at line 108 of the template — `p ? Object.assign({role: p.role || 'member'}, p) : {role: 'member'}` — is preserved; per saved memory, breaking it causes stuck-load.

### Console disposition

[apps/console/](../../apps/console/) is a 65-line admin stub. Skipped in Phase A — no iframe wrapper, no nav entry. Either folded into a future RM admin page during V3.43 or deleted outright in Phase C; decision deferred until the admin/role-management RM page is scoped.

### Service worker

[sw.js](../../sw.js) bumps `CACHE_VERSION` 14 → 15. The 33-entry `APP_URLS` precache list is dropped along with the `caches.open(APPS_CACHE).then(addAll)` install step. The `APPS_CACHE` constant + runtime stale-while-revalidate routing for `/apps/*` (line 213) **stay** — iframe loads still benefit from runtime caching during Phase A. Phase C deletes the constant when `/apps/` itself goes.

### Public-site nav prune

Three orphaned `Lab Apps` `<li>` entries removed from [index.html](../../index.html) (drawer, desktop, mobile sheet — IDs `nav-apps`, `dnav-apps`, `more-apps`). The `nav-apps`/`dnav-apps`/`more-apps` toggle in [user-system.js:661](../../user-system.js#L661) is dropped so the IDs stay hidden if any orphan markup survives. The `#dnav-rm`/`#nav-rm`/`#more-rm` conditional-on-login entries stay — that's the single sanctioned entry to RM. `#dnav-dashboard`/`#dnav-admin` are left in place; they get cleaned up in Phase C once the SPA routes are deleted.

### Firestore rules

Confirmed no-op. All 23 collections used by the 12 lab apps already have rule blocks at [firestore.rules:189-466](../../firestore.rules#L189-L466). No deploy needed in Phase A.

## Overlap audit (port targets for Phase B)

For each duplicate pair, **the RM page wins**. The audit below names the lab-app features the RM page is missing — those are the Phase B port-time merge targets. RM-only features stay; lab-app duplicates of those get dropped.

### activity-tracker (V3.49)

The existing [rm/pages/activity-tracker.html](../../rm/pages/activity-tracker.html) is an iframe wrapper — there is **no native RM tracker yet**. The full 2,100-line lab app must be ported wholesale.

- **Port targets:** daily entry logging, hierarchical category tree (6 main → 25+ leaves), category manager UI, weekly summary + analytics charts, Web Speech API input, calendar import + auto-suggest, Huddle plan integration, AI co-pilot suggestions, bulk paste, mobile swipe nav.
- **Firestore:** `trackerData/{uid}`, `trackerEntries/{uid}/entries/{entryId}` — both **user-scope** (`shadowJson: false`). Schema fields: `date, time, category, text, duration, source (manual|speech|huddle|calendar|ai), isHuddle, huddleId`.

### settings (V3.48)

The lab-app and RM page are **orthogonal**, not duplicates. Port both into one consolidated page rather than choosing one.

- **Lab-app side (port to RM):** profile (name, bio, photo), notification master toggle, per-app push toggles (6 apps), quiet hours, `shareActivity` privacy toggle, calendar OAuth UI, per-app admin links (meetings, chat, equipment), test notification sender, push diagnostics, app-version checker, logout.
- **RM-only (preserve):** connection registry — external integrations (email forwarding, calendar subscriptions, repo links, messaging, finance) with type tabs and sortable table.
- **Firestore:** lab-app reads `userSettings/{uid}` (**user-scope, shadowJson: false**), `users/{uid}` (existing), `meetingConfig/settings` + `chatConfig/settings` (admin-only role detection). RM uses `settings/connections.json` — different file, different schema, no conflict.

### compliance (V3.46)

RM page is richer (IRB/IACUC protocol management). Lab app contributes only file-upload UI, which RM already has via the Student Training tab.

- **Port targets:** essentially nothing — the lab-app form is a strict subset. The Student Training tab in [rm/pages/compliance.html](../../rm/pages/compliance.html) already reads `complianceSubmissions`. Confirm at port time that all lab-app form fields (`type, title, completionDate, expirationDate, certificateUrl`) are exposed in the RM training-submit flow; if not, add them.
- **RM-only (preserve):** IRB/IACUC protocol management (title, protocol number, PI, approval/expiration/renewal dates, related proposal link, notes), three-tab UI (IRB/IACUC/Student Training), deadline chip, PI verify button.
- **Firestore:** both write `complianceSubmissions` with same schema — `submittedBy, submitterName, type, title, completionDate, expirationDate, certificateUrl, status, createdAt, updatedAt`. RM also manages a separate `compliance/{docId}` collection for protocols (lab-scope).

### procurement → receipts (V3.45)

RM page is a superset (982 lines vs 337). Lab app contributes the lightweight student-submission form.

- **Port targets:** confirm RM's submission form covers every lab-app field — vendor, orderNumber, sourceType (pcard|ilab|amazon_business|vendor_po|internal), project, fundingAccount, date, lineItems\[\], ilabUrl, notes. The existing RM submission tab covers most; add any missing fields.
- **RM-only (preserve):** parsed-receipt tab (Claude PDF parsing), budget analysis dashboards (by account/category/project), inventory integration on receive, receipt status lifecycle (`needs_receipt → needs_submission → submitted → approved → reimbursed`).
- **Firestore shape diff:** lab-app uses `fundingAccount` (string id); RM expects `account_number` (string number). RM uses per-user subcollection `financeReceipts/{userId}/{docId}` for receipts vs lab-app's flat `procurement` collection. Port-time decision: keep `procurement` as the student-submission inbox; map to `financeReceipts` on PI review.

### purchases → purchase-requests (V3.44)

RM page (256 lines) adds the admin-review surface that the 126-line lab app lacks.

- **Port targets:** confirm RM submission form covers all lab-app fields — itemDescription, vendor, estimatedCost, justification, urgency (routine|needed_soon|urgent), category (supplies|equipment|software|other).
- **RM-only (preserve):** all-requests review (not user-filtered), Pending vs Resolved tabs, sortable table with urgency chip, approve/deny/mark-received actions, inline edit, "Receive & create inventory item" flow, PI notes field, pagination.
- **Firestore:** both use `purchaseRequests`. Schema agrees on all submission fields; RM extends `status` enum beyond `requested` (adds `received`, `approved`, `denied`).

### inventory (V3.47)

Lab app is a 48-line stub. RM page (903 lines) is the real thing — 8 tabs, 30+ subcategories, multi-location stock, search/filter/sort, in-line editing, live sync, "Create Group" feature.

- **Port targets:** none. Delete `/apps/inventory/`.
- **Firestore:** RM uses `inventory/{docId}` with kind discriminator. Lab app has zero schema.

## Key decisions

- **Iframe bridge first, native port second.** Lower risk: every app keeps working through the entire migration. Each Phase B port only touches one app and is independently shippable.
- **RM page wins on duplicates.** RM's IndexedDB cache + LIVE_SYNC + Firestore-only adapter is the architectural target; the lab apps' direct-Firestore + iframe-postMessage pattern is being retired. The audit above lists the lab-app features that must be merged in before the lab-app version is deleted.
- **Public site keeps its login flow.** Per the locked decisions, the public-site `#/login` route stays; only the **Apps** dropdown is removed. The `#dnav-rm` link to RM is the sole post-login surface visible from the public nav.
- **`Teaching` and `Service` folded into Research.** Keeps the top nav at six items rather than eight. Both pages stay reachable by URL; only their nav entry shape changes.
- **`APPS_CACHE` constant kept (for now).** Iframe loads still benefit from runtime stale-while-revalidate of `/apps/*` files. Phase C drops the constant when `/apps/` itself goes.
- **No firestore.rules deploy in Phase A.** Per the audit, every collection already has the right rule shape. Per-app rule touch-ups happen in Phase B if and only if a port adds a new field a rule depends on.
