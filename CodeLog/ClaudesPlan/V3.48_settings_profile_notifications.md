# V3.48 — Settings: Profile + Notifications tabs

## Goal

Pull the lab-app `/apps/settings/` Profile + Notifications sections
into the existing RM settings page at
[rm/pages/settings.html](../../rm/pages/settings.html). Settings
becomes a one-stop shop for self-service configuration: edit your
display name + bio, manage push-notification preferences, and (still)
maintain the connection registry. Calendar Integration + admin
diagnostics + version checker stay in `/apps/settings/` for now —
those pull in the deferred CalendarService OAuth flow (V3.51) or are
admin-only tooling that doesn't justify a port until the rest of the
app retires.

| Lab-app section | Status in V3.48 |
|---|---|
| Profile (name, bio, photo, shareActivity) | ✅ ported to RM Profile tab |
| Notifications (master toggle, per-app push, quiet hours) | ✅ ported to RM Notifications tab |

| App Version checker | skip — RM auto-refreshes via service-worker bump |
| Logout | skip — RM has the firebridge profile dropdown |

The two new tabs sit at the front of the existing tab bar:

```
[ Profile ] [ Notifications ] [ All Connections (N) ] [ Email (n) ] [ Calendars (n) ] ...
```

`+ Add Connection` button is hidden when Profile or Notifications is
the active tab (it only makes sense on the connection sub-tabs).

## Data layout

Two existing user-scope writes are exercised — both already covered
by [firestore.rules](../../firestore.rules):

- **`users/{uid}`** — Profile tab writes `name`, `bio`, `shareActivity`,
  `updatedAt`. Read by every RM page that resolves a member's display
  name. Rule: owner write at line 96.
- **`userSettings/{uid}`** — Notifications tab writes a nested
  `notifications` object: `{ enabled, quietHours: { enabled, start, end },
  apps: { chat, meetings, equipment, huddle, scheduler, activityTracker
  }: { push: bool } }`. Same shape the lab app at
  [/apps/settings/app.js:300-316](../../apps/settings/app.js#L300-L316)
  uses, so existing Cloud Function notification triggers
  (`functions/helpers/notify.js`) keep filtering correctly. Rule:
  owner read/write at line 85.

The connection-registry tab continues to use the existing
`settings/connections.json` route (per-user via api-routes.js).

## Implementation

### `rm/js/settings.js`

Reorganized around an `activeTab` dispatcher. `loadAndRender()` now:

1. Renders the tab bar with Profile, Notifications first, then the
   existing connection sub-tabs (counts shown only on connection
   tabs).
2. Hides `#add-item` when Profile / Notifications is active.
3. Dispatches to `renderProfile(content)`, `renderNotifications(content)`,
   or `renderConnections(content, connections)`.

`renderProfile()` does a fresh `firebridge.db().collection('users').doc(uid).get()`
read so off-page edits show up, displays an avatar (photo or initial),
and writes `{ name, bio, shareActivity, updatedAt }` on save with a
surgical `set(..., { merge: true })`.

`renderNotifications()` reads `userSettings/{uid}` (with a default
fallback if the doc doesn't exist), renders 6 per-app push toggles +
quiet-hours time pickers, dims the time row when DnD is off, and
saves a single nested `notifications: { ... }` patch via `set` with
merge.

Both saves show a small inline status pill ("Saved" / "Save failed:
...") and surface errors to the console. No LIVE_SYNC — settings
are per-user, single-tab in practice.

The existing `editItem` / `deleteItem` / `+ Add Connection` flow on
the connection tabs is preserved verbatim. `add-item.onclick`
defensively checks `_isConnectionTab()` so a stale handler from a
previous render doesn't fire when the button has been hidden.

### `rm/css/settings.css`

New page-scoped stylesheet (~140 LOC). Covers `.settings-card`,
`.settings-toggle-row`, the iOS-style `.settings-switch` (CSS-only
checkbox toggle), `.settings-status` pills, `.settings-time-row` for
the quiet-hours pickers, and the avatar styling. Uses RM CSS
variables — no aliasing needed since this isn't a port-from-lab CSS.

### `rm/pages/settings.html`

Added `<link rel="stylesheet" href="/rm/css/settings.css">` directly
after the global stylesheet. No other markup changes needed —
`#tabs`, `#content`, `#add-item` already exist.
