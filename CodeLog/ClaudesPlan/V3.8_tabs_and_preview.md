# V3.8 — Tabs, Preview Toggle, and Nested Widgets

## Goal

Three architectural changes to the course builder:
1. **Admin/public view toggle** — preview the class page as students see it
2. **Named tabs** — admin creates, names, and manages tabs as top-level navigation
3. **Widgets inside sections** — widgets can only be added within sections, not as standalone blocks

## Current State

- Flat `layout` array of blocks (sections and widgets as peers at top level)
- Three-column builder: left palette (sections) + canvas + right palette (widgets)
- No preview mode — admin always sees editing view
- No tab navigation — single page of stacked blocks

## New Data Model

```javascript
tabs: [
  { id: 'tab_abc', name: 'Overview', sections: [
    { key: 'overview', id: 'sec_1', widgets: [
      { kind: 'text', id: 'wgt_1', content: '...' }
    ] }
  ] },
  { id: 'tab_def', name: 'Resources', sections: [
    { key: 'files', id: 'sec_2', widgets: [] }
  ] }
]
```

- **Tabs** are the top-level navigation containers
- **Sections** (from `SECTION_REG`) are containers within a tab — each has native content (text editor, file manager, or scheduler)
- **Widgets** (from `WIDGET_REG`) are content blocks nested inside sections — text, image, video, links, embed, divider
- Section keys are unique across all tabs (content stored in shared `_classData` fields)
- Widget content stored inline in the tab structure

## New UI

### Tab bar
- Horizontal tab buttons at top of class page content
- Admin: click to switch, double-click to rename, × to delete, + button to add
- Public: click to switch, no editing controls

### Admin editing view
- "Add Section" dropdown below tab bar
- Sections stack vertically in active tab
- Each section: chrome bar (handle, label, ↑/↓, ×) + native content + nested widgets + "Add Widget" dropdown
- Widgets inside sections: mini chrome bar (label, ↑/↓, ×) + widget content
- No side palettes — replaced by inline dropdowns

### Preview toggle
- "Preview" button in header actions area
- Toggles between admin editing view and public read-only view
- Button text changes to "Edit" when in preview mode

## Migration

- If Firestore doc has `tabs` field → use directly
- If `layout` field → sections become tab sections, following widgets attach to preceding section, wrap in single "Home" tab
- If only `sections` array → create section objects with empty widgets, wrap in single "Home" tab

## Approach

### Phase 1: Data model + migration
- Replace `_layout` with `_tabs`, `_activeTabId`, `_previewMode`
- `migrateLegacy()` converts old formats to tabs
- `tabsToSections()` derives flat sections array for backwards compat
- `findWidgetById()` helper to search active tab's sections' widgets

### Phase 2: Preview toggle
- `_previewMode` boolean flag
- `togglePreview()` switches between editing and read-only render
- `isEditing()` helper: `_viewType === 'admin' && !_previewMode`
- All renderers use `isEditing()` instead of `_viewType === 'admin'`

### Phase 3: Tab system
- Tab bar UI with add/rename/delete
- `switchTab()` gathers DOM content, changes active tab, refreshes canvas
- `addTab()`, `deleteTab()`, `renameTab()` manage tab lifecycle
- Cannot delete the last tab

### Phase 4: Nested widgets
- Sections rendered with chrome + body + widgets area + "Add Widget" button
- Widgets rendered with mini chrome + body, nested inside sections
- `addSection()` adds to active tab; `addWidget(sectionId)` adds to specific section
- `removeSection()`, `moveSection()`, `removeWidget()`, `moveWidget()` operate within tabs
- DnD for section reordering; arrow buttons for widget reordering

### Phase 5: CSS updates
- New: tab bar, section container, nested widget, add widget bar, preview banner
- Remove: three-column palette grid, palette items
- Keep: drop indicator, modal, autosave status, block content styles

## Key Decisions

- **Sections unique across tabs**: A section key (e.g., "overview") can only appear once across all tabs since its text content is stored in a single `_classData` field.
- **No side palettes**: Replaced with inline dropdowns for cleaner UX and better mobile support.
- **DnD for sections, arrows for widgets**: Nested DnD zones are too complex; use arrow buttons for widget reordering within sections.
- **Tab rename via prompt()**: Simpler than inline contenteditable, less error-prone.
- **Cannot delete last tab**: Always need at least one tab.
- **Backwards compatibility**: `migrateLegacy()` handles old `layout` and `sections` formats automatically.

## Risks

- **Speakers section complexity**: Scheduler component expects `#cb-speakers-container`. Works since the container is inside a section block — ID is preserved.
- **File data pre-loading**: Must scan all tabs to find file-type sections. Done in `wireClassPage()`.
- **Tab switching loses unsaved content**: Mitigated by calling `gatherContentFromDOM()` before switching tabs.
