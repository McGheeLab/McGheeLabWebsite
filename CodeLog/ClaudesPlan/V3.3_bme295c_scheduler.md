# V3.3 — Section-Based Class Builder & Speaker Scheduler

## Goal
Transform the class system from a single-purpose scheduler into a configurable class builder. Each class has selectable sections (overview, syllabus, schedule, speakers, files, simulations, lectures, homeworks, exams). The speaker scheduler is preserved as one component type within this framework. Build BME 295C as the first class using overview + speakers + files sections.

## Architecture

### Section Registry Pattern
```javascript
const SECTION_REG = {
  overview:    { label: 'Overview',        component: 'text',    field: 'overviewContent' },
  syllabus:    { label: 'Syllabus',        component: 'text',    field: 'syllabusContent' },
  schedule:    { label: 'Class Schedule',  component: 'text',    field: 'scheduleContent' },
  speakers:    { label: 'Guest Speakers',  component: 'speakers' },
  files:       { label: 'Files',           component: 'files',   path: 'files' },
  simulations: { label: 'Simulations',     component: 'text',    field: 'simulationsContent' },
  lectures:    { label: 'Lecture Notes',   component: 'files',   path: 'lectures' },
  homeworks:   { label: 'Assignments',     component: 'files',   path: 'homeworks', hasDue: true },
  exams:       { label: 'Exams',           component: 'files',   path: 'exams',     hasDue: true }
};
```

### Three Component Types
1. **`text`** — Admin: textarea editor with save + live preview. Student: read-only rendered content.
2. **`files`** — Upload to Firebase Storage (`classes/{classId}/{section}/`), metadata in `classFiles` Firestore collection. Admin+speaker upload, students download. Optional due date for homeworks/exams.
3. **`speakers`** — Full dual-mode scheduler: sessions (checkbox voting, greedy bipartite assignment) + freeform (drag-to-select, HSL heatmap). Invite key access, talk submission, optimization.

### Three Role-Based Views
- **Admin** — Full edit access to all sections, settings panel, view switcher to preview as speaker/student
- **Speaker** — Availability voting, talk submission, file access (upload to designated sections)
- **Student** — Read-only view of all published content and confirmed speaker schedule

### Module State
```
_activeSection, _viewType, _classData, _speakers, _currentSpeaker,
_useKeyAuth, _expanded, _scheduleId, _adminViewMode, _previewSpeakerIdx
```
State cached at module scope — section switching dispatches via `switchSection()` without re-fetching Firestore.

## Data Model

### Firestore: `schedules`
```
{
  id: "bme295c",
  title, subtitle, semester, description,
  sections: ["overview", "speakers", "files"],  // ← NEW: which sections enabled
  overviewContent: "...",                        // ← NEW: text section content
  syllabusContent: "...",                        // ← text section content
  scheduleContent: "...",                        // ← text section content
  simulationsContent: "...",                     // ← text section content
  mode: "sessions" | "freeform",
  startDate, endDate, slotDefs, startHour, endHour, granularity,
  ownerUid
}
```

### Firestore: `classFiles` (NEW)
```
{
  classId: "bme295c",
  section: "files" | "lectures" | "homeworks" | "exams",
  fileName, fileUrl, storagePath, fileSize, fileType,
  description, dueDate,
  uploadedBy, uploadedByName,
  createdAt
}
```

### Firestore: `participants` (unchanged)
```
{
  key, scheduleId, speakerName, speakerEmail, speakerUid,
  talkSummary, questions, presentationLink,
  availability, assignedSlot,
  createdAt, updatedAt
}
```

## Page Flow
```
#/classes/bme295c → renderClassPage("bme295c") → header + placeholder
  → wireClassPage("bme295c") → load schedule + speakers from Firestore
  → determine view type (admin/speaker/student)
  → buildClassLayout(sections) → sidebar nav + content area
  → switchSection(firstSection) → dispatch to component type
    → text: buildTextSection / wireTextSection
    → files: buildFilesSection / wireFilesSection
    → speakers: buildSpeakersSection → admin/speaker/student sub-view
```

## Key Decisions
- **Section registry** — single source of truth maps section keys to component types; easy to add new section types
- **Three reusable components** — text, files, speakers cover all 9 section types with shared code
- **Module-level state** — avoids re-fetching Firestore on section switch; data loaded once on page entry
- **Settings as a section** — admin toggles sections via checkboxes; stored as `sections` array on schedule doc
- **File metadata in Firestore** — `classFiles` collection with storage URL, size, description, optional due date; actual files in Firebase Storage
- **Preserved scheduler** — all existing scheduler code wrapped inside the speakers component dispatcher
- **Generic collections** — `schedules` + `participants` + `classFiles` — reusable for any class
