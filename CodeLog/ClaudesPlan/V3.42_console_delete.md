# V3.42 — Console: delete the stub

## Goal

Resolve the `/apps/console/` admin-stub disposition deferred from V3.40. Per
the V3.40 audit, the lab app is a 65-line "Coming Soon" placeholder with no
data layer, no Firestore reads/writes, no functionality beyond rendering
section titles for an unbuilt admin panel. Delete it outright rather than
porting an empty shell.

## Reorder note

V3.42 was originally allocated to scheduler in the V3.41 plan. Scheduler is
genuinely heavy — even a subset port (My Schedulers tab only) requires
bringing in the 1,382-LOC `/scheduler.js` engine and rewiring the
`schedules` + `participants` Firestore CRUD. Doing it justice needs a
dedicated session. Bumped scheduler later in the queue and slotted the
trivial overlap-app deletions (V3.42–V3.46) here so the migration keeps
visible momentum without bundling complexity. Final remainder order:

## What's in the stub

[apps/console/app.js](../../apps/console/app.js) (65 LOC) renders four
hard-coded "section" cards (App Management, User Permissions, Integrations,
Usage & Logs) with description text and an `Under Development` badge. No
data layer, no event handlers (`wire()` is empty), no Firestore reads/writes,
no Cloud Function calls. It auth-walls non-admins via
`McgheeLab.AppBridge.isAdmin()`.

Phase A (V3.40) had already opted not to create a `rm/pages/app-console.html`
iframe wrapper — the wrapper would have been a placeholder for a
placeholder. The Phase A nav (Operations group) deliberately omits a Console
entry. So the deletion has zero nav impact.

## Approach

1. `git rm -r apps/console/` — drops `app.js` (65 LOC), `index.html`,
   `styles.css`. The pre-existing local cache-buster modifications to
   `apps/console/index.html` (`?v=2` query params) are discarded with the
   directory.
2. Remove the `console` entry from the `LAB_APPS` registry in
   [lab-apps.js](../../lab-apps.js) (lines 53–65). The registry is already
   orphaned in the public nav (V3.40 dropped the Apps menu) but consistency
   matters when Phase C deletes the file entirely.
3. `sw.js` `CACHE_VERSION` 16 → 17.
4. CHANGELOG + ARCHITECTURE updates.

## Future: where the admin functions actually go

The four "section" titles in the stub map to existing or in-flight RM
features, not a future replacement for the console:

- **App Management** — N/A. RM doesn't have per-app feature flags; nav
  gating in [rm/js/nav.js](../../rm/js/nav.js) handles "show/hide app".
- **User Permissions** — covered by the future `rm/pages/admin.html`
  scoped in the V3.40 plan (role admin: gate `admin`, edits `users/{uid}`).
  Built alongside the public-site editors port (V3.43–V3.45 of the V3.40
  plan); now slotted later since this V3.42 reordered the deletions
  ahead of it.
- **Integrations** — already exists at [rm/pages/settings.html](../../rm/pages/settings.html)
  (connection registry: email forwards, calendar subs, repo links).
- **Usage & Logs** — partly covered by [rm/pages/api-usage.html](../../rm/pages/api-usage.html)
  for Anthropic API calls; broader audit logging is not currently built.

No follow-up work needed for V3.42 itself — the stub never delivered any
of the functions it promised, and the actual surfaces are accounted for.
