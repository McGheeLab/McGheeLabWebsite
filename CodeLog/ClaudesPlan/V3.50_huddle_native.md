# V3.50 — Huddle: native RM port (subset; Lab Schedule deferred)

## Goal

Replace the V3.40 iframe wrapper at
[rm/pages/app-huddle.html](../../rm/pages/app-huddle.html) with a
native renderer at [rm/pages/huddle.html](../../rm/pages/huddle.html).
Same lift-and-patch playbook as V3.41 meetings + V3.49 activity-tracker:
copy the lab `app.js` (2,888 LOC), surgically patch the bootstrap +
container id + db helper + a couple of AppBridge references, leave the
2,800 LOC of view + Firestore subscription code unchanged.

The Lab Schedule view (`teamavail` section — Team Availability Gantt,
recurring availability templates, per-week overrides) depends on
`apps/shared/schedule-service.js` + `apps/shared/calendar-service.js`,
both of which are V3.51-deferred (the Calendar OAuth refactor moves
those out of `sessionStorage` tokens into `oauthConnections/{uid}`).
The renderer already null-checks every `McgheeLab.ScheduleService` /
`McgheeLab.CalendarService` call (e.g. `?.resolveScheduleForUser`,
`?.resolveAvailabilityOverlap`) so the section degrades gracefully —
empty Gantt, no recurring templates — until V3.51 lands.

## Patches applied to the lifted file

1. **`window.McgheeLab = window.McgheeLab || {}`** at top — same
   pattern as V3.49. Optional-chaining lookups for `MobileShell?` /
   `CalendarService?` / `ScheduleService?` still need the namespace
   to exist; otherwise it's a `ReferenceError` rather than `undefined`.
2. **Container id `#app` → `#huddle-root`** — avoids collisions with
   other RM pages.
3. **`db()` prefers `firebridge.db()`** when available, falls back to
   `firebase.firestore()`.
4. **Bootstrap rewrite** — replaces the dual AppBridge / direct-Firebase
   path (lines 85-155 of the lab app) with the RM contract:
   `firebridge.gateSignedIn()` → `await firebridge.whenAuthResolved()`
   → `_user = firebridge.getUser()`, `_profile = firebridge.getProfile()`
   → `loadSettings()` → `render()` → `subscribePlans/Help/Rundown/RundownConfig`.
   `CalendarService.init` and `ScheduleService.init` stay guarded by
   `if (McgheeLab.X)` so they no-op without the services. Mobile
   `enableTabSwipe` stays guarded too. Profile-role flips trigger
   re-render via `firebridge.onAuth`.
5. **`notifyResize()` AppBridge guard** — the lab app's `notifyResize`
   called `McgheeLab.AppBridge.isEmbedded()` non-defensively. Patched
   to check `McgheeLab.AppBridge && McgheeLab.AppBridge.isEmbedded`
   first so the function no-ops cleanly when AppBridge is undefined
   (which it always is in RM — there's no parent window to message).

The remaining ~2,800 LOC of plan-feed + help-requests + rundown +
team-availability rendering, Firestore subscriptions, drag-to-create
plan grid, and category/project pickers transfers verbatim.

## Key decisions

- **Verbatim lift.** 2,800 LOC of view + Firestore subscription code
  is correct and battle-tested; rewriting it to fit a "different" RM
  idiom would introduce regressions. Same call I've made for every
  large port in this migration.
- **Lab Schedule degrades, not stubbed.** The renderer already
  null-checks every CalendarService / ScheduleService call. Patching
  in mock services here would just create more code to delete in
  V3.51. Easier to leave the optional chaining and let V3.51 drop
  the real services into place.
- **No LIVE_SYNC.attach on the page.** The renderer already uses
  Firestore `onSnapshot` directly via `subscribePlans` /
  `subscribeHelp` / `subscribeRundown` / `subscribeRundownConfig`,
  storing unsubscribers in `_unsubscribe` / `_unsubHelp` /
  `_unsubRundown` / `_unsubRundownConfig`. Layering LIVE_SYNC over
  that would duplicate the work; the existing approach is fine.
- **`huddleSettings/{userId}` stays user-scope.** The lab app reads
  the signed-in user's own huddle defaults; firestore.rules already
  restricts it to `request.auth.uid == userId`. No widening needed.

## Verification

Local dev:

```bash
cd /Users/alexmcghee/Documents/GitHub/McGheeLabWebsite
RM_WRITE_ALLOWLIST=mcgheealex@gmail.com python3 rm/server.py
```

Smoke flow:

1. Sign in as guest → pending-access overlay covers the page.
2. Sign in as lab member → page renders with the Plan Feed tab active.
3. Open Plan Feed → time-grid for the current week renders. Drag to
   create a new plan; popup form opens. Save → plan appears live.
4. Switch to Help tab → existing help requests render. Submit a new
   request → appears live.
5. Switch to Rundown tab → category-grouped task list. Add a task
   with a project tag.
6. Switch to Lab Schedule (`teamavail`) → empty Gantt (V3.51 will
   populate it once ScheduleService loads). Console silent.
7. Switch to My Plans → own plans grouped by week.
8. Switch to Settings (lab member view) → user's huddleSettings doc.
9. `http://localhost:8000/rm/pages/app-huddle.html` → meta-refreshes
   to `/rm/pages/huddle.html`.
10. `http://localhost:8000/apps/huddle/` → standalone fallback still
    loads (Lab Schedule works there until V3.51).
11. Open the same URL in a second tab → mutate in tab 1 → tab 2's
    `onSnapshot` listeners fire and the UI updates within ~1 sec.
12. DevTools → Application → Cache Storage → only `mcgheelab-*-v25`
    caches.
