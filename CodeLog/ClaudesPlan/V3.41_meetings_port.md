# V3.41 — Meetings: native RM port (Phase B begins)

## Reorder note

The V3.40 plan listed scheduler as the first Phase B port (V3.41) and meetings
second (V3.42). On audit, scheduler turned out to depend on three shared
modules — ScheduleService (418 LOC, Firestore subscriptions for
`huddleScheduleTemplates` / `huddleScheduleOverrides` / `scheduleCustomEvents`),
CalendarService (536 LOC, Google + Outlook OAuth with sessionStorage tokens),
and the parent `scheduler.js` engine (1,382 LOC). Porting it natively in V3.41
would have either (a) front-loaded the OAuth-to-Firestore refactor that the
V3.40 plan deliberately deferred to V3.51 (equipment), or (b) shipped a
feature regression on the My Schedule tab.

Meetings (1,892 LOC) is materially cleaner: only `meetings`, `meetingConfig`,
and the already-RM-cached `users` collection. No shared services beyond an
optional `MobileShell` call. **Swapped V3.41 ↔ V3.42:** meetings is the
proof-of-concept, scheduler is now V3.42.

## Goal

Replace the Phase A iframe bridge for /apps/meetings/ with a native RM page
that uses the api.load + IndexedDB cache + LIVE_SYNC contract. After this
release, [rm/pages/meetings.html](../../rm/pages/meetings.html) is the source
of truth; [rm/pages/app-meetings.html](../../rm/pages/app-meetings.html) is a
`<meta http-equiv="refresh">` redirect for bookmark survival.

## Approach

### Routes ([rm/js/api-routes.js](../../rm/js/api-routes.js))

Two new lab-scope routes registered between the existing `lab/users.json` and
the `tasks/` block. Both Firestore-only from day one (the lab app wrote
direct), so `shadowJson` is off (default).

```js
api.registerRoute('meetings/list.json', {
  scope: 'lab', collection: 'meetings', wrapKey: 'meetings',
  cache: MEDIUM,
});
api.registerRoute('meetings/config.json', {
  scope: 'lab', collection: 'meetingConfig', doc: 'settings',
  cache: LONG,
});
```

The audited collections (`meetings`, `meetingConfig`) and the already-cached
`lab/users.json` route give the renderer everything it needs.

### Renderer ([rm/js/meetings.js](../../rm/js/meetings.js))

1,827-LOC port of [/apps/meetings/app.js](../../apps/meetings/app.js). Lifted
near-verbatim with three structural patches:

- **Bootstrap** — replaces the dual AppBridge / `firebase.auth().onAuthStateChanged`
  handshake with `firebridge.gateSignedIn() → firebridge.whenAuthResolved() →
  loadAndRender() → LIVE_SYNC.attach()`. Exposes the live handle on `_live` so
  surgical writes can set `_live.suppressUntil`.
- **Data load** — single `loadAndRender()` calls `api.load('meetings/list.json')`,
  `api.load('meetings/config.json')`, and `api.load('lab/users.json')` in
  parallel. `_meetings` sorts ascending by date (matches the lab app's
  `orderBy('date','asc')`); `_config` is the single-doc data; `_labMembers`
  filters out `role === 'guest'` and sorts by name.
- **Live updates** — `LIVE_SYNC.attach({paths: [...3 paths], refresh:
  loadAndRender, tag: 'meetings'})`. Replaces three separate `onSnapshot`
  listeners; the api adapter routes the subscriptions through the same
  Firestore listeners but de-duplicates and lets us suppress echoes.
- **Writes** — `saveConfig`, `createMeeting`, `updateMeeting`, `deleteMeeting`
  call `firebridge.db().collection(...)` directly (surgical, mirroring
  [rm/js/receipts.js](../../rm/js/receipts.js) lines 79–96). Each write
  bumps `_live.suppressUntil = Date.now() + 2500` so the LIVE_SYNC echo
  doesn't trigger a redundant `loadAndRender`. After the await resolves the
  write mutates `_meetings`/`_config` in-memory and calls `renderMain()` so
  the user sees the change instantly, not after the suppress window expires.
- **MobileShell + iframe-resize calls dropped** — RM is the parent window;
  there's no `mcgheelab-app-resize` postMessage to fire, and the top-nav
  lives in the page rather than via `MobileShell.configure`.

The toast helper falls back to RM's global `toast()` ([rm/js/toast.js](../../rm/js/toast.js))
when present, otherwise renders an `.mtg-toast` element directly (matches
the lab-app behavior so existing CSS animations work).

### Page ([rm/pages/meetings.html](../../rm/pages/meetings.html))

Standard RM contract: `runtime-mode → firebase SDK (incl. storage) →
firebase-bridge → profile-bootstrap → local-cache → util →
api-firestore-adapter → api-routes → live-sync-helper → toast → sync-status →
nav`, then `meetings.js`. The `<main>` container is `<div id="mtg-root">` —
the renderer queries this rather than `#app` to avoid colliding with other RM
pages that may use `#app`. `firebase-storage-compat` is included because the
presentation-files modal uses Firebase Storage for uploads.

### Styles ([rm/css/meetings.css](../../rm/css/meetings.css))

Lifted from [/apps/meetings/styles.css](../../apps/meetings/styles.css)
(1,203 LOC) plus the `.app-card`/`.app-btn`/`.app-input`/`.app-label`/
`.app-empty`/`.app-badge` classes from
[/apps/shared/app-base.css](../../apps/shared/app-base.css). The lab base
used CSS vars (`--muted`, `--accent`, `--danger`, `--success`, `--warning`,
`--surface2`, `--radius-lg`) that don't exist in
[/rm/css/style.css](../../rm/css/style.css). A single `body { ... }` block
at the top aliases each to the RM equivalent (`--text-muted`, `--primary`,
`--red`, `--green`, `--amber`, `#eef2f6`, `12px`) so every selector below
renders correctly without per-rule rewrites.

### Nav flip ([rm/js/nav.js](../../rm/js/nav.js))

Operations → Meetings now points at `/rm/pages/meetings.html` (was
`/rm/pages/app-meetings.html`). The wrapper file at
[rm/pages/app-meetings.html](../../rm/pages/app-meetings.html) is replaced
with a `<meta http-equiv="refresh">` redirect so users with bookmarks pinned
to the Phase A URL still land in the right place. Phase C will delete the
redirect alongside the rest of the `app-*.html` wrappers.

### firestore.rules

No-op. The collections used by the renderer (`meetings` at line 317,
`meetingConfig` at line 325, `users` already gated by `isLabMember()`) all
have rule blocks that match the audit's read/write pattern.

### sw.js

`CACHE_VERSION` 15 → 16. Standard bump for every code change.

## Key decisions

- **Reorder over scope-cut.** Doing scheduler in V3.41 would have bled into
  V3.51's OAuth refactor or shipped a feature regression. Meetings being
  self-contained is the right proof-of-concept.
- **Lift renderer near-verbatim.** The lab app's UX is the target; only the
  data layer + bootstrap change. This minimizes diff and preserves visual
  parity. Phase B style cleanup (folding into RM's data-table / modal
  conventions) is a follow-up if and when the user wants it.
- **Surgical writes via firebridge.db().** Matches the receipts.js pattern
  and avoids the 80-doc-collection rewrite cost of `api.save()` for a single
  agenda-item toggle. The `_live.suppressUntil` echo guard is the same
  mechanism used by receipts.
- **Optimistic in-memory mutation + renderMain after each write.** Without
  this the UI stalls 2.5s waiting for the LIVE_SYNC suppress window to
  close. The trade-off is that a failed Firestore write would leave a stale
  optimistic value on screen until the next snapshot fires; for meetings
  data (low conflict rate, friendly users) this is acceptable.
- **No MobileShell shim.** RM ships its own top nav and full-page chrome.
  The MobileShell phone shell from /apps/shared/ is a lab-app concept that
  doesn't apply here. The mobile media query at the bottom of meetings.css
  still gives a vertical sidebar collapse, which is the only mobile
  affordance the renderer actually depended on.
- **/apps/meetings/ kept on disk through Phase B.** Standalone URL still
  works as a fallback during the migration window. Phase C deletes it.

## Verification

Local dev:

```bash
cd /Users/alexmcghee/Documents/GitHub/McGheeLabWebsite
RM_WRITE_ALLOWLIST=mcgheealex@gmail.com python3 rm/server.py
# open http://localhost:8000/rm/pages/meetings.html
```

Smoke checks:

- Sign in as guest → pending-access overlay covers the page (firebridge gate).
- Sign in as lab member → page renders with sidebar (Next Meeting / Schedule
  / Archive / My Items; Settings appears only for `role: 'admin'`).
- Console silent (no `[meetings] ...` warnings, no Firestore permission
  errors).
- DevTools → Application → IndexedDB → `rm-cache` shows entries for
  `route::meetings/list.json`, `route::meetings/config.json`,
  `route::lab/users.json` after first load.
- Add an agenda item → reload → item persists (proves api.load reads back
  surgical writes through the cache).
- Open same URL in second tab, mutate in tab 1 → tab 2 re-renders within
  ~1s after the suppress window closes (proves LIVE_SYNC fan-out).
- `http://localhost:8000/rm/pages/app-meetings.html` → meta-refreshes to
  `/rm/pages/meetings.html`.
- `http://localhost:8000/apps/meetings/` → still loads standalone (Phase A
  fallback intact until Phase C).
- DevTools → Application → Cache Storage shows only `mcgheelab-*-v16`.
